# Spark Mall 后续开发路线图

生成时间：2026-05-27  
规划目标：从当前“可演示/可联调原型”推进到“可上线项目”。  
约束说明：本路线图仅规划后续开发，不修改前端页面。

## 项目当前状态总览

### 已完成部分

| 模块 | 当前状态 |
| --- | --- |
| 前端页面 | 首页、商品、盲盒、盒柜、订单、寄售、用户中心、积分、余额、消息等主要页面已存在 |
| 前端接口封装 | 主业务接口集中在 `src/api/*`，大部分页面已接入 alova 请求 |
| 后端服务 | 已有独立 `server/`，Express + TypeScript，提供统一响应结构 |
| 基础接口联调 | 首页、商品、盲盒、订单、盒柜、寄售、用户、地址、消息等页面调用接口基本覆盖 |
| Supabase 基础 schema | 已有商品、分类、盲盒、订单、盒柜商品、寄售、banner、用户等基础表 |
| 本地演示数据 | 后端有内置 mock 数据，Supabase 有 seed 数据，能支撑演示链路 |

### 待开发部分

| 模块 | 待开发内容 |
| --- | --- |
| 真实登录 | 手机号/短信/微信登录、token/session、用户身份校验 |
| 订单系统 | 订单状态机、金额重算、订单归属校验、取消/超时/发货/售后 |
| 支付系统 | 支付单、微信支付、余额支付、回调验签、幂等、退款 |
| 库存系统 | 商品库存、奖池库存、库存锁定、扣减、回滚 |
| 盲盒逻辑 | 权重抽奖、保底、奖池、抽奖记录、并发控制 |
| 地址系统 | 地址表、真实 CRUD、默认地址逻辑 |
| 账户流水 | 余额流水、积分流水、提现记录、寄售收益结算 |
| 文件上传 | OSS/Supabase Storage/对象存储接入，头像和商品图上传 |
| 后台管理 | 商品、订单、用户、发货、寄售、提现、banner 管理 |

### 待优化部分

| 模块 | 优化内容 |
| --- | --- |
| 后端结构 | `server/src/app.ts` 单文件过大，需要拆 route/controller/service/repository |
| 前端请求体系 | alova、uni.request、旧 service/mock 并存，需要统一 |
| 字段模型 | 数据库 snake_case、后端 mapper、前端 camelCase 需要建立稳定契约 |
| 接口语义 | 下单、支付、地址、寄售等写操作大量使用 GET，需要改为 POST/PUT/DELETE |
| 错误处理 | 缺少业务错误码、统一异常、用户提示和日志关联 ID |
| 类型质量 | API 类型不完整，部分参数/返回值为 any |
| 测试体系 | 缺少接口测试、核心业务单元测试、端到端冒烟测试 |

### 高风险部分

| 风险 | 当前表现 | 上线要求 |
| --- | --- | --- |
| 支付风险 | 支付接口目前是假支付，直接更新订单状态 | 必须接真实支付与回调验签 |
| 登录风险 | 固定 token、固定用户、无真实鉴权 | 必须实现真实身份认证和权限隔离 |
| 库存风险 | 无库存字段和原子扣减 | 必须防止超卖和重复出奖 |
| 抽奖风险 | 未按真实权重/库存/保底抽奖 | 必须可审计、可复现、可并发 |
| 数据安全 | service role key 有暴露风险，后端绕过 RLS | 必须轮换密钥、隔离环境变量、补服务端鉴权 |
| 订单资金风险 | 余额、积分、寄售收益无可信流水 | 必须建立账户流水和事务 |

## 第一阶段：让项目稳定运行

目标：把当前演示链路整理成稳定可联调状态，消除明显接口缺口、字段不一致和假数据断点，让项目能在本地/测试环境稳定跑通。

| 任务 | 优先级 | 预计工作量 | 是否必须 |
| --- | --- | --- | --- |
| 梳理所有页面实际调用接口，形成接口清单和字段契约 | P0 | 1-2 天 | 是 |
| 对齐前端 `src/api/*` 与后端返回字段，补齐缺失字段默认值 | P0 | 2-3 天 | 是 |
| 修复 mock/Supabase 两种模式下返回结构不一致的问题 | P0 | 2-3 天 | 是 |
| 将 `server/.env`、根 `env/.env`、部署环境变量职责划清 | P0 | 0.5-1 天 | 是 |
| 补齐地址、消息、余额、积分、推广、提现等假数据，使页面不空、不报错 | P1 | 2-4 天 | 是 |
| 建立统一业务错误码和错误响应格式 | P1 | 1-2 天 | 是 |
| 给后端接口增加基础参数校验和必填校验 | P1 | 2-3 天 | 是 |
| 清理或标注旧 `mock-server/`、`src/service/app/*`、旧 mock service 的使用边界 | P2 | 1-2 天 | 否 |
| 恢复本地 `pnpm type-check`、lint、build 可执行环境 | P1 | 0.5-1 天 | 是 |
| 输出接口联调验收表：页面、接口、参数、返回、状态 | P1 | 1 天 | 是 |

阶段完成标准：

- 本地前端和后端能稳定启动。
- 主流程页面不因字段缺失报错。
- mock 模式和 Supabase 模式返回结构一致。
- 已知假实现被明确标注，不再与真实功能混淆。

## 第二阶段：完善核心业务

目标：把商城最核心的订单、库存、盲盒、盒柜、寄售逻辑做成可信业务闭环，为真实支付和上线打基础。

| 任务 | 优先级 | 预计工作量 | 是否必须 |
| --- | --- | --- | --- |
| 设计订单状态机：待支付、已支付、待发货、已发货、已完成、已取消、售后中等 | P0 | 1-2 天 | 是 |
| 重构订单创建逻辑，服务端重算商品金额、运费、积分、优惠 | P0 | 3-5 天 | 是 |
| 新增库存字段/库存表，支持库存锁定、扣减、释放 | P0 | 4-6 天 | 是 |
| 建立盲盒奖池模型：奖品、权重、库存、等级、保底规则 | P0 | 4-7 天 | 是 |
| 抽奖接口改为事务或数据库 RPC，保证并发下不重复扣库存 | P0 | 4-7 天 | 是 |
| 新增抽奖记录表，记录用户、订单、奖池、奖品、随机结果、时间 | P0 | 2-3 天 | 是 |
| 盒柜商品状态流转规范化：待开、可提货、寄售中、已出售、已提货、已分解 | P0 | 2-3 天 | 是 |
| 寄售流程补齐商品归属校验、价格校验、取消寄售、售出结算 | P1 | 4-6 天 | 是 |
| 提货流程补齐收货地址、运费、发货状态、物流单号 | P1 | 3-5 天 | 是 |
| 分解流程接入积分流水，避免前端本地直接加减积分 | P1 | 2-3 天 | 是 |
| 补订单/库存/抽奖/寄售接口测试 | P1 | 3-5 天 | 是 |

阶段完成标准：

- 下单、支付前订单、库存锁定、取消释放库存有明确规则。
- 盲盒抽奖按奖池和库存执行，结果可追溯。
- 盒柜、寄售、提货、分解状态不再靠前端假设。
- 核心资金和权益变动都能在后端记录。

## 第三阶段：真实业务接入

目标：接入真实第三方能力，让登录、支付、短信、上传等从“模拟可用”变为“生产可用”。

| 任务 | 优先级 | 预计工作量 | 是否必须 |
| --- | --- | --- | --- |
| 确定认证方案：自建 JWT、Supabase Auth 或微信小程序登录体系 | P0 | 1-2 天 | 是 |
| 接入微信小程序 code2Session/openId/session_key | P0 | 2-4 天 | 是 |
| 接入手机号短信验证码服务，完成发送、校验、频控、防刷 | P0 | 3-5 天 | 是 |
| 后端所有用户相关接口改为从 token/session 获取 user_id | P0 | 3-5 天 | 是 |
| 接入微信支付，创建支付单、预支付、前端调起支付 | P0 | 5-8 天 | 是 |
| 接入支付回调，完成验签、金额校验、幂等处理、订单状态更新 | P0 | 4-6 天 | 是 |
| 接入余额支付，建立余额流水和事务扣款 | P0 | 3-5 天 | 是 |
| 接入退款能力或至少预留退款单模型 | P1 | 3-5 天 | 视上线范围 |
| 接入 OSS/Supabase Storage，用于头像、商品图、详情图上传 | P1 | 2-4 天 | 是 |
| 替换测试图片和 `picsum.photos` 为正式素材/对象存储资源 | P1 | 2-4 天 | 是 |
| 接入短信、支付、OSS 的异常监控和失败重试策略 | P1 | 2-3 天 | 是 |
| 编写第三方服务配置文档和沙箱/生产切换说明 | P1 | 1-2 天 | 是 |

阶段完成标准：

- 用户身份真实可靠，接口能按用户隔离数据。
- 支付必须通过第三方回调确认，不能由前端或普通接口直接置已支付。
- 短信、微信、OSS 等配置可在测试/生产环境独立切换。
- 关键第三方失败有明确错误提示和日志。

## 第四阶段：上线准备

目标：完成安全、部署、监控、日志、数据初始化、回滚和验收流程，让项目具备正式发布条件。

| 任务 | 优先级 | 预计工作量 | 是否必须 |
| --- | --- | --- | --- |
| 轮换并清理所有已暴露或疑似暴露的密钥 | P0 | 0.5-1 天 | 是 |
| 建立环境变量规范：local、test、staging、production | P0 | 1-2 天 | 是 |
| 后端禁止生产环境使用 mock fallback | P0 | 1 天 | 是 |
| Supabase RLS/权限复核，确认 service role 仅在后端私密环境使用 | P0 | 1-2 天 | 是 |
| 增加接口鉴权、角色权限、资源归属校验审计 | P0 | 2-4 天 | 是 |
| 增加请求日志、错误日志、支付回调日志、订单关键操作日志 | P0 | 2-4 天 | 是 |
| 接入监控告警：接口错误率、支付失败、库存异常、抽奖异常 | P1 | 2-4 天 | 是 |
| 准备数据库迁移流程、seed 策略、备份与回滚方案 | P0 | 2-3 天 | 是 |
| 建立发布流程：构建、部署、健康检查、回滚 | P0 | 2-3 天 | 是 |
| 压测核心接口：登录、商品列表、下单、支付回调、抽奖 | P1 | 2-4 天 | 是 |
| 做上线前安全检查：密钥、CORS、上传限制、参数校验、防重放、防刷 | P0 | 2-3 天 | 是 |
| 编写上线验收清单和运维手册 | P1 | 1-2 天 | 是 |
| 准备客服/运营后台最小功能或人工 SQL 操作规范 | P2 | 3-7 天 | 视运营要求 |

阶段完成标准：

- 生产环境无 mock 数据和占位配置。
- 关键资金、订单、库存、抽奖链路有日志、监控和回滚方案。
- 密钥和环境变量隔离清晰。
- 数据库迁移、部署、回滚流程可重复执行。

## 推荐总体排期

| 阶段 | 建议周期 | 是否可并行 |
| --- | --- | --- |
| 第一阶段：稳定运行 | 1-2 周 | 前后端可并行 |
| 第二阶段：核心业务 | 3-5 周 | 订单/库存/盲盒可部分并行，但需统一模型 |
| 第三阶段：真实接入 | 3-5 周 | 登录、支付、OSS 可并行，但支付依赖订单模型 |
| 第四阶段：上线准备 | 1-2 周 | 可与第三阶段后半段并行 |

保守估算：达到可上线状态约 **8-14 周**，取决于支付、微信登录、库存/抽奖规则复杂度，以及是否需要同步开发后台管理系统。

## 上线前必须完成清单

- 真实登录和用户数据隔离。
- 真实支付、支付回调、验签和幂等。
- 订单金额服务端重算。
- 库存锁定、扣减、释放。
- 盲盒奖池、权重、库存和抽奖记录。
- 余额/积分/寄售收益流水。
- 地址真实落库。
- 生产环境禁用 mock fallback。
- 密钥轮换与环境变量隔离。
- 核心接口日志、监控、备份、回滚。

## 不建议优先投入的事项

| 事项 | 原因 |
| --- | --- |
| 大规模重做前端 UI | 当前主要风险在后端真实业务闭环，不在页面视觉 |
| 过早开发复杂营销系统 | 优惠券、活动、分销应在订单/支付/账户稳定后再做 |
| 直接上线当前支付逻辑 | 目前是假支付，存在严重资金风险 |
| 继续扩展旧 mock service | 会增加新旧体系混乱，应逐步统一到正式 API |
| 先做后台大而全 | 可先做最小运营后台或临时管理工具，避免拖慢核心上线 |
